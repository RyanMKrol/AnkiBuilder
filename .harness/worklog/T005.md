# T005 — Translation stage via local claude -p

Date: 2026-07-13

## What I did
- Added `src/translate/runClaude.js`: default runner wrapping a real `claude -p <prompt>`
  subprocess call (spawnSync), throwing on non-zero exit or spawn error.
- Added `src/translate/index.js`: `translateCorpus(corpus, { runClaude })` iterates
  corpus items, builds a structured prompt per item (asks for JSON
  `{target, pronunciation, hint?}`), calls the injected `runClaude` (defaults to the
  real CLI wrapper), and parses the JSON response.
  - Items with an existing `notes` (e.g. an EPUB-extracted candidate translation) still
    call the model (to verify/get pronunciation) but keep `notes` as the authoritative
    `target` rather than blindly regenerating it.
  - Malformed/non-JSON model output, or JSON missing required string fields, is caught
    and reported in a separate `errors` array (`{ id, error }`) — the item is skipped,
    not written as an invalid card.
  - Returns `{ cards, errors }` where `cards` is validated via T002's `validateCards`
    before returning.
- Added `test/translate/index.test.js` with a mock `runClaude` (no real `claude` call):
  untranslated item → target+pronunciation merged into a valid card; hint passthrough;
  already-translated item's `notes` preserved as `target` even when the mock model
  returns a different translation; malformed model output for one item surfaces as an
  `errors` entry while the other item in the same corpus still translates successfully.
