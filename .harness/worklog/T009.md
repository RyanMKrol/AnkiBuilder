# T009 — CLI orchestrator over the run directory (resumable)

Date: 2026-07-13

## What I did

Added `src/cli/index.js` (`runCli(argv, deps)`, dependency-injectable for testing) and
`src/cli/bin.js` (shebang entrypoint), wired via a new `bin.anki-builder` field in
`package.json`. Four subcommands drive the pipeline over a run directory (paths from
`runPaths` in T002):

- `assemble --run <dir> [--template <name> | --epub <path> --lang <code>]` → `corpus.json`
  (T003/T004)
- `translate --run <dir>` → `cards.json` (T005)
- `audio --run <dir> --voice <id>` → enriches `cards.json` with `audio` filenames, copying
  generated files from the ElevenLabs cache (`stateHome()/audio/<voice>`) into the run's
  `audio/` dir (T006)
- `deck --run <dir> [--name <name>]` → `deck.apkg` (T007)

Each command is resumable: it checks whether its output artifact already exists (or, for
`audio`, whether every card already has its audio file present) and skips redoing the work,
just logging a "reusing" message.

`.env` loading: `bin.js`'s shebang is `#!/usr/bin/env -S node --env-file-if-exists=.env`
(Node's native flag, no new dependency), so `ELEVENLABS_API_KEY` etc. from a `.env` in the
working directory land in `process.env` before `runCli` executes; when no `.env` is present
the flag is a no-op and existing exported env vars are used as-is.

The default `fetchTts` implementation calls the real ElevenLabs TTS endpoint
(`POST /v1/text-to-speech/:voiceId`) via `globalThis.fetch` — only exercised in production
use; tests always inject a mock.
